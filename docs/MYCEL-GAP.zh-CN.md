# Mycel 想补上的空白

状态：说明文档

这份文档说明：为什么过去比较少出现像 Mycel 这样的协议，以及 Mycel 想处理的问题在今天通常是怎么被处理的。

## 1. 为什么过去很少有像 Mycel 这样的协议

Mycel 想处理的是一个比较尴尬的中间层问题：

- 比普通文档协作更严格
- 比区块链更不想依赖全网单一共识
- 比 Git 更在意读者侧的默认采用结果
- 比一般应用后端更在意可验证历史和可移植性

这类需求一直存在，但通常不够集中，不足以自然催生出一个独立的协议市场，所以过去大多被拆开处理。

### 1.1 这个问题是分散的，不是某个行业里的集中痛点

大多数用户一开始真正需要的是：

- 好用的编辑器
- 文档版本管理
- 中心化后台
- 网页协作

很少有人一开始就同时要求：

- 可验证历史
- 多个有效分支并存
- 默认阅读结果由固定规则导出
- 还要支持去中心化复制

### 1.2 现有方案各自只吃掉一部分需求

因此，市场通常不会自然逼出一个像 Mycel 这样的中间层协议。

### 1.3 这种协议很难用一句话讲清楚

Git 很好解释，区块链也比较容易描述。

但 Mycel 这种“不是 Git，不是链，而是一个面向可验证文本历史和受规则约束阅读结果的协议”，并不容易一眼让人理解。

### 1.4 工程成本不低，但短期商业回报不明显

要做出 Mycel 这种系统，需要同时处理：

- canonicalization
- signed objects
- replay
- accepted-state rules
- partial replication
- wire sync

这类工作量不小，但又不像聊天应用、支付系统或内容平台那样容易快速商业化。

## 2. 今天这些问题通常怎么处理

### 2.1 中心化平台

最常见的做法是使用：

- Google Docs
- Notion
- Confluence
- 各类 CMS 和知识库平台

这类系统通常：

- 由平台数据库保存全部状态
- 由平台决定默认版本
- 可以看历史，但通常没有独立可验证、可移植的历史模型

优点：

- 好上手
- 上线快
- 学习成本低

限制：

- 平台依赖强
- 历史很难独立验证
- 分支和治理模型通常比较弱

### 2.2 Git

另一种常见做法是把文本当成 Git 仓库来管理。

这样可以得到：

- branch
- commit history
- merge
- review flow

优点：

- 很适合作者和维护者
- 历史保留能力强
- 多分支模型成熟

限制：

- 更像作者工具，不一定适合一般读者
- “默认阅读结果如何导出”不是它的核心设计目标
- app、governance 和 accepted-state 的语义不太容易自然表达

### 2.3 区块链或 smart contract

有些系统会把重要状态写到链上，再用链上共识决定唯一状态。

优点：

- 一致性强
- 可验证性高
- 有统一的全局状态

限制：

- 太重
- 成本高
- 对很多文本治理场景来说属于过度设计
- 很多情况下其实并不需要全网只有一个唯一真版本

### 2.4 定制后端和数据库

现实里最常见的做法，其实还是自己写一个应用后端：

- 状态存在数据库里
- 审计依赖 audit log
- 权限、阅读结果和历史都放在应用后端里实现

优点：

- 最实际
- 最容易接入已有产品
- 上线最快

限制：

- 通常不具备互操作性
- 可移植性差
- 几乎没有协议层验证

## 3. Mycel 想补的是哪一层空白

Mycel 想补上的，是中心化平台和全局共识链之间的那一层：

- 历史应该可验证
- 默认采用结果应该可解释
- 多个有效分支应该允许并存
- 但又不想承担区块链式全局共识的成本

换句话说，Mycel 想提供的是：

- 可验证历史
- 由固定规则导出的默认阅读结果
- 去中心化复制
- 但不要求全网收敛成唯一真版本

它不是要替代所有现有方案，而是要把这些零散需求重新组织成一套分层、可规范、可互操作的协议设计。

## 4. 一句话版本

过去不是没有这类需求，而是它一直被中心化平台、Git、区块链和定制后端用不完整的方式各自吸收掉了。Mycel 的价值，在于把这些分散需求重新组合成一套可验证、可导出默认阅读结果、又能去中心化复制的协议。
