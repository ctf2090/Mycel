Protocol-first 文字系統

文本歷史、受治理閱讀狀態與去中心化複製。

Mycel 是一個適合長期文本、評註系統與受治理參考文本集合的協議棧。 它把可重播驗證的歷史、治理訊號與去中心化複製拆成不同層次，同時保留彼此互通。

預設閱讀版本不是全網共識，而是依固定 profile 規則，從已驗證物件推導出的結果。

開啟 Repository 閱讀規格
目前狀態

`v0.1` spec-first 專案。目前工作集中在 first-client 範圍收斂、deterministic verification、 replay 行為，以及 simulator 驗證流程。

它不是什麼

它不是 blockchain，不是 Git 複製品，也不是泛用檔案傳輸層。Mycel 針對的是有治理需求的文字系統。

今天能做什麼

Rust CLI 已能支援 repo-local validation、object inspection、accepted-head inspection、 report inspection 與 deterministic simulator runs。

為什麼是 Mycel

多數分散式協作系統要嘛把狀態集中在平台，要嘛為程式碼協作或全域共識最佳化。 Mycel 走的是更窄但更明確的路徑：text-first 歷史、由 profile 決定的 accepted reading， 以及帶有明確 replay 與 verification 邊界的 replication。

可驗證歷史

revision 預期可以從 canonical objects replay、驗證與重建。

受治理的閱讀狀態

accepted head 來自固定 selector 規則與已驗證的 View objects。

容許分叉

多個有效 head 可以並存，不需要把整個網路壓成一個全域真相。

今天就能試什麼

現在的 CLI 還不是 production client 或 node，但已經適合在本 repo 內做驗證與 simulator 流程。

如果你是從全新環境開始，請先看 Dev Setup。

檢查 Workspace

cargo run -p mycel-cli -- info

驗證 Fixtures

cargo run -p mycel-cli -- validate fixtures/object-sets/minimal-valid/fixture.json --json

執行 Simulator

cargo run -p mycel-cli -- sim run sim/tests/three-peer-consistency.example.json --json

支持或一起推進 Mycel

我們希望與正在處理受治理文字系統、評註層或長期參考文本集合的設計夥伴、grant 計畫與早期協作者合作。 現在的支持會直接幫助我們完成第一個可互通 client 的核心：verification、replay、可重建儲存，以及由 profile 決定的 accepted reading。
