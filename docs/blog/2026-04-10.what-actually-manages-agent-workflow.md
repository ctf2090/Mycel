When people ask how we manage multiple coding agents in one repository, they
often expect a hidden supervisor runtime.

They imagine one central process deciding who exists, what each agent should do
next, when a task is "really done," and how every follow-up should be routed.

What we actually use is simpler and more legible:

- a small set of repo-native scripts
- tracked Markdown rules
- local registry and mailbox files
- explicit Git boundaries

That choice is deliberate.

We wanted the coordination layer to stay inspectable in the repository itself,
not trapped inside one long chat or one opaque orchestration service.

## The Goal Is Durable Coordination, Not Invisible Magic

The operational problem is not only "how do we prompt multiple agents?"

It is:

- how do we know who currently owns a slice
- how do we start and end work in a way other sessions can trust
- how do we leave continuation context that survives interruption
- how do we keep Git history aligned with actual work boundaries
- how do we notice when planning surfaces are drifting behind landed work

These are workflow questions, not just model questions.

That is why the Mycel coordination layer is mostly built from small scripts and
files with narrow jobs.

## The Main Surfaces We Actually Use

At a high level, our agent workflow relies on a few concrete surfaces:

- `AGENTS.md` and `AGENTS-LOCAL.md` define the shared operating rules
- `.agent-local/agents.json` records agent identity, role, lifecycle state, and mailbox path
- `.agent-local/mailboxes/*.md` hold durable handoff entries
- `.agent-local/agents/<agent_uid>/checklists/` keeps per-agent checklist copies
- `scripts/` contains the thin tools that move those surfaces forward

The important design point is that each surface is visible and inspectable.

A new or resumed session does not need to infer everything from chat memory.
It can read the repo-local artifacts and reconstruct enough state to continue
safely.

## Bootstrap And Identity

The first coordination problem is simple: who is this agent, and what role does
it hold right now?

We handle that with:

- `scripts/agent_bootstrap.py`
- `scripts/agent_registry.py`

`agent_bootstrap.py` is the fast-path wrapper. It claims a role, records the
model id, sets up the per-agent checklist copies, and starts the tracked work
cycle for that chat.

`agent_registry.py` is the lower-level lifecycle tool behind that flow. It owns
the durable identity model: stable `agent_uid`, human-facing `display_id`,
role, scope, mailbox path, and lifecycle state such as `active`, `inactive`,
or `paused`.

This matters because "who is currently working?" should never depend on which
browser tab still happens to be open.

## Workcycles Instead Of One Endless Session

We do not treat agent work as one vague continuous state.

We use:

- `scripts/agent_work_cycle.py begin`
- `scripts/agent_work_cycle.py end`

Those commands give each user-command cycle a clear boundary.

The begin step marks the agent active, creates the next checklist copy, and
emits a canonical timestamp line.

The end step checks closeout requirements, validates mailbox state, records the
after-work timestamp, and refuses a "clean finish" when important obligations
are still missing.

That sounds procedural because it is procedural.

But the payoff is that "in progress" stops being a fuzzy feeling. It becomes a
tracked unit of work with a start, a finish, and evidence about whether the
cycle actually closed cleanly.

## Handoffs That Survive Chat Loss

If chat memory were enough, interruption would not be a serious problem.

In practice, sessions get compacted, disconnected, or resumed later by someone
else. That is why we keep handoffs in mailbox files and use:

- `scripts/mailbox_handoff.py`

The mailbox is not just a status dump. It is the continuation surface another
agent can trust.

A useful handoff records:

- current state
- scope
- evidence
- next suggested step
- whether the entry is still open, resolved, or superseded

This is one of the places where "workflow tooling" matters more than raw model
memory. A very capable agent still cannot recover context cleanly if the only
record is an ambiguous chat transcript.

## Git Safety Is Part Of The Coordination Layer

Our workflow tools also manage Git boundaries directly.

The goal is not only to commit code. The goal is to make the commit boundary
match the owned work item.

The key scripts here are:

- `scripts/agent_safe_commit.py`
- `scripts/agent_push.py`

`agent_safe_commit.py` stages only the explicit allowlisted paths for the work
item, aborts when unrelated staged files are present, and adds commit trailers
that preserve agent identity plus model metadata.

`agent_push.py` pushes the specific agent commit instead of pushing a branch by
habit, which helps keep the landing action intentional and auditable.

This is a good example of the broader design philosophy:

- do not leave important coordination behavior to convention alone
- make the boundary executable

## Checklists And Planning Refresh

Some workflow problems are not about who owns a file right now. They are about
whether the surrounding planning surfaces are still aligned with reality.

For that layer we use:

- per-agent checklist copies generated under `.agent-local/agents/...`
- `scripts/item_id_checklist_mark.py`
- `scripts/check-plan-refresh.py`

The checklist copies turn tracked rules into local execution state without
editing the tracked source documents.

`check-plan-refresh.py` asks a practical question: after enough landed work,
are roadmap, implementation checklist, issue surfaces, or web surfaces now
stale enough that doc sync should happen?

That may sound secondary compared with coding.
It is not.

If the coordination layer only manages current execution and never notices
planning drift, teams still lose trust in the repo as a shared source of truth.

## Why We Prefer Small Scripts Over A Hidden Orchestrator

There are real tradeoffs here.

This approach creates more visible process.
It asks agents to touch multiple explicit surfaces.
It does not feel as sleek as one "smart" runtime that claims to manage
everything behind the curtain.

But it has three advantages we care about:

- the state stays inspectable in the repository
- the pieces can fail independently without making the whole workflow opaque
- teams can tighten or swap one primitive without rewriting the whole stack

In other words, we are not trying to hide coordination.
We are trying to make it durable.

## Takeaway

The workflow is not managed by one secret master agent.

It is managed by a small repo-native coordination layer:

- registry for identity
- workcycle boundaries for lifecycle
- mailbox handoffs for continuation
- Git guardrails for ownership
- planning checks for drift

That stack is less glamorous than agent hype.
It is also much closer to what real multi-agent repository work actually needs.
