# Mycel的價值

這份文件把 Mycel 的價值主張改寫成一種較容易對外溝通的方式：
不是先講協定特性，而是先從人們今天使用現存工具時已經感受到的痛點切入。

## 核心方向

與其先說 Mycel 是什麼，不如先說：

- 為什麼今天很多工具明明能協作，卻仍然讓人無法放心保存重要文本
- 為什麼你看得到版本，卻不一定看得到「這個結果是怎麼形成的」
- 為什麼多版本一出現，閱讀體驗就會變亂，或最後被硬壓成單一版本

Mycel 想補上的，就是這些痛點之間長期缺的一層。

## 現存應用痛點 vs Mycel

| 現存應用痛點 | 常見感受 | Mycel 的對應價值 | 可用說法 |
| --- | --- | --- | --- |
| 歷史存在，但不容易驗證 | 我看得到版本紀錄，但很難確定這個結果怎麼來的。 | 可重播、可驗證的文本歷史 | 不只是有版本，而是歷史可以被重建與檢查。 |
| 預設版本是平台狀態，不是可解釋結果 | 現在看到這一版，但我不知道為什麼是這一版。 | 預設閱讀版本由固定規則導出 | 預設版本不是黑箱，也不是管理員手動拍板。 |
| 分歧常被硬壓成單一版本 | 不同看法只能被覆蓋，最後只剩一個官方答案。 | 允許多個有效分支並存 | 不假裝世界上只有一個真版本。 |
| 分叉存在時，閱讀體驗就會變亂 | 一旦出現多版本，就不知道預設該讀哪個。 | 在保留分歧下仍可導出 accepted reading | 可以同時容納分歧與預設閱讀。 |
| 平台把編輯、治理、展示全綁在一起 | 資料在哪裡、怎麼顯示、誰說了算，全都綁在同一套產品裡。 | 把歷史、治理訊號與複製拆層但保持互通 | 資料不必等於平台。 |
| 離開原平台後，語境與規則很難保留 | 匯出內容可以，但匯不出判斷邏輯。 | 規則與物件關係可以被保留與重建 | 帶走的不只是內容，還有結果如何形成。 |
| 協作工具擅長編輯，不擅長長期保存 | 短期寫作方便，但不適合做長期文化或知識保存。 | 面向長期文本、評註與參考文本集合 | 為長期可驗證保存而設計，而不只是日常協作。 |
| 共識型系統太重，文件型工具又太輕 | 區塊鏈太重，文件平台又不夠可信。 | 位在中心化協作平台與全域共識系統之間 | 補上兩者之間缺的那一層。 |
| 很難把治理訊號變成可檢查的輸入 | 誰支持哪個版本常常散落在會議、聊天室或平台權限裡。 | 把治理訊號當成可驗證物件處理 | 治理不再只是口頭或後台操作。 |
| 工具常逼你接受單一產品 worldview | 產品怎麼定義內容、版本與權限，我們就得跟著接受。 | 協定核心保持中立，語意放在 profile 與 app layer | 不把單一應用邏輯寫死進底層。 |

## 三個最強訊息

如果要濃縮成最能對外講的三句：

1. 不是只有歷史，而是可驗證的歷史。
2. 不是只有多版本，而是分歧存在時仍有可解釋的預設閱讀。
3. 不是把資料鎖進平台，而是讓歷史、治理與複製可以分開存在。

## 建議的對外敘事順序

### 1. 先講痛點

現有工具不是沒有版本，而是版本結果通常不可驗證、不可解釋。

### 2. 再講 Mycel

Mycel 讓文本歷史可重播，讓預設閱讀可推導，讓資料不綁死在單一平台。

### 3. 最後講適用場景

它特別適合：

- 長期文本
- 評註系統
- 治理型知識庫
- 文化保存
- 受治理的參考文本集合

## 可直接使用的對外說法

### 版本 A：痛點導向

現有工具要嘛把歷史藏在平台狀態裡，要嘛只能告訴你「現在是什麼」，卻很難告訴你「為什麼會變成這樣」。Mycel 想補上的，不是又一個協作介面，而是一套可重播驗證的文本歷史、可檢查的預設閱讀結果，以及不依賴單一平台的複製基礎。

### 版本 B：治理導向

在多方協作的文本系統裡，真正困難的不是編輯，而是如何在保留分歧的同時，仍然導出一個可預期、可解釋的預設版本。Mycel 把這件事從平台黑箱變成可驗證、可推導的規則結果。

### 版本 C：文化保存導向

對長期保存的文本來說，「現在看到的版本」不夠，還需要知道它如何形成、誰提出了哪些治理訊號，以及這些結果是否能在別處重建。Mycel 為這類需求提供可驗證的歷史、依規則導出的閱讀結果，與去中心化複製。
